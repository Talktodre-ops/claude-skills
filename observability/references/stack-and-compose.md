# The stack and its Compose wiring

Real config, generalized names. Rename the network, the internal DNS zone
(`*.internal.example`), and the project label to the target project. The whole
stack runs as Docker Compose on one host.

## Components

| Role | Image | Port (host:container) | Notes |
|---|---|---|---|
| Metrics store | `prom/prometheus:v2.53` | 9090:9090 | TSDB on local disk, 30d retention |
| Log store | `grafana/loki:3.1` | 3100:3100 | object store on S3, distroless (no shell) |
| Query pane | `grafana/grafana-oss:11.1` | 3001:3000 | host 3001 avoids a dev server on 3000 |
| Collector + RUM | `grafana/alloy:v1.3` | 12347:12347 | docker log tail and the Faro receiver |
| Host metrics | `prom/node-exporter:v1.8` | 9100:9100 | CPU, memory, disk, network of the box |
| Container metrics | `gcr.io/cadvisor/cadvisor:v0.49` | 8082:8080 | per-container CPU, memory, IO, needs privileged |
| Error backend | `glitchtip/glitchtip:v4.1` | 8090:8000 | Sentry-API-compatible |
| Error DB | `postgres:16-alpine` | internal | GlitchTip's own database |
| Error queue | `valkey/valkey:8.0` | internal | GlitchTip's own Redis, isolated from the app cache |

Pin every tag to a real release. The source system ran almost everything on
`latest`, which is not reproducible: a box rebuild can silently pull a new major
version of Loki or Grafana and break the schema or the provisioning. Pin, and bump
on purpose.

GlitchTip needs four containers: a one-shot `migrate` (run on first boot,
`restart: "no"`), the `web` server, a `worker`, plus its own Postgres and Redis.
Keep that Redis separate from your application cache.

## The shared network

Every observability service and the application's own containers join one external
Docker network, so they resolve each other by service name:

```yaml
# in each compose file
networks:
  obs-net:
    external: true     # created once: docker network create obs-net
```

Grafana reaches Prometheus at `http://prometheus:9090` and Loki at
`http://loki:3100` over this network. Cross-host callers (an app on ECS, the
browser RUM proxy) cannot use Docker DNS, so they use internal DNS names like
`loki.internal.example:3100` instead. Keep the two namespaces straight: service
names for same-host, internal DNS for cross-host.

Name the long-running services `restart: unless-stopped`. Give the stateful ones
(Prometheus, Loki, Grafana, the error DB) named volumes.

Optional split: keep host and container exporters (node-exporter, cAdvisor) in a
second compose file so you can start and stop host-metrics independently of the
core stack. If you do, do not also define them in the core file under the same
container names, or the two files collide on a `docker compose up`.

## Prometheus

```yaml
# prometheus/prometheus.yml
global:
  scrape_interval: 15s
  evaluation_interval: 15s
  external_labels: { env: "${PROMETHEUS_ENV:-local}", project: "myapp" }

scrape_configs:
  - job_name: prometheus
    static_configs: [{ targets: ["localhost:9090"] }]

  - job_name: loki
    static_configs: [{ targets: ["loki:3100"] }]

  - job_name: node-exporter
    static_configs: [{ targets: ["node-exporter:9100"] }]
    scrape_interval: 30s

  - job_name: cadvisor
    static_configs: [{ targets: ["cadvisor:8080"] }]
    scrape_interval: 30s
    metric_relabel_configs:                # drop high-cardinality cAdvisor series
      - source_labels: [__name__]
        regex: 'container_(tasks_state|spec_.*|cpu_load_average_10s)'
        action: drop

  # App tasks with rotating IPs, discovered from a file the box regenerates.
  - job_name: app-prod
    file_sd_configs:
      - files: ['/opt/monitoring/prometheus/targets/app.json']
        refresh_interval: 30s
    metrics_path: /metrics
    relabel_configs:
      - { target_label: env, replacement: production }
```

Retention is a process flag, not in this file: `--storage.tsdb.retention.time=30d`.
The file service-discovery target list is regenerated by a small script on a timer
(see `instrumentation.md`); when the file is absent (local dev) the job just yields
no targets, which is harmless.

Two cautions. The Prometheus admin API (`--web.enable-admin-api`) is unauthenticated,
so only enable it on a private box. Watch the `job` label: a `file_sd` entry can
inject its own `job` label that overrides `job_name`, and if your alert rules query
one name while the series carries the other, the alerts silently never fire.
Confirm the label your series actually has.

## Loki

Loki indexes label sets, not content, so the config is mostly about the schema and
retention. Use one config for local (filesystem) and one for production (S3),
selected by an env var, and enable env expansion (`-config.expand-env=true`).

```yaml
# loki/loki.s3.yaml  (production)
auth_enabled: false                 # single tenant; put real auth in front if multi-tenant
server: { http_listen_port: 3100 }
common:
  replication_factor: 1             # single node
  ring: { kvstore: { store: inmemory } }
  storage:
    s3:
      region: ${LOKI_AWS_REGION}
      bucketnames: ${LOKI_S3_BUCKET}    # auth via instance role, no keys in file
      insecure: false
schema_config:
  configs:
    - from: "2024-01-01"
      store: tsdb
      object_store: s3
      schema: v13
      index: { prefix: index_, period: 24h }
limits_config:
  retention_period: 720h            # 30 days
  ingestion_rate_mb: 16
  ingestion_burst_size_mb: 32
  max_streams_per_user: 10000       # the cardinality guard
compactor:
  retention_enabled: true           # the compactor enforces retention
  compaction_interval: 10m
  retention_delete_delay: 2h
  delete_request_store: s3
analytics: { reporting_enabled: false }
```

The local variant is identical except `object_store: filesystem` and a
`filesystem: { chunks_directory: /loki/chunks }` block, so a developer runs the
same stack without S3. Loki's distroless image has no shell, so it has no Compose
healthcheck; depend on it by start order, not health.

Set a second retention backstop at the storage layer (an S3 lifecycle rule a few
days past the Loki retention) in case the compactor falls behind.

## Grafana

Grafana is configured entirely by `GF_*` environment variables, and provisioned
from a mounted directory.

```yaml
# compose service: grafana
environment:
  GF_SECURITY_ADMIN_USER: admin
  GF_SECURITY_ADMIN_PASSWORD: ${GRAFANA_ADMIN_PASSWORD}
  GF_USERS_ALLOW_SIGN_UP: "false"
  GF_SERVER_ROOT_URL: ${GRAFANA_ROOT_URL:-http://localhost:3001}
  GF_ANALYTICS_REPORTING_ENABLED: "false"
  GF_ANALYTICS_CHECK_FOR_UPDATES: "false"
  # SMTP for alert email
  GF_SMTP_ENABLED: "true"
  GF_SMTP_HOST: ${SMTP_HOST}
  GF_SMTP_USER: ${SMTP_USER}
  GF_SMTP_PASSWORD: ${SMTP_PASSWORD}
  # let the CloudWatch datasource use the instance role (no static keys)
  GF_AWS_ALLOWED_AUTH_PROVIDERS: "default,keys,credentials,ec2_iam_role"
volumes:
  - ./grafana/provisioning:/etc/grafana/provisioning:ro
depends_on:
  prometheus: { condition: service_healthy }
  loki: { condition: service_started }
```

The provisioning directory (datasources, dashboards, alerting) is the subject of
`grafana-as-code.md`. Mount it read-only so the files stay the source of truth.

## Grafana Alloy (the collector)

Alloy replaces Promtail. It reads the Docker socket, tails container logs, derives
the label taxonomy, drops the monitoring stack's own logs, and also runs the Faro
receiver for browser RUM. River syntax:

```alloy
// discover running containers from the Docker socket
discovery.docker "containers" {
  host             = "unix:///var/run/docker.sock"
  refresh_interval = "30s"
}

// derive low-cardinality labels from container metadata, drop the obs stack itself
discovery.relabel "containers" {
  targets = discovery.docker.containers.targets
  rule { source_labels = ["__meta_docker_container_name"] target_label = "service" regex = "/(.*)" replacement = "$1" }
  // map service name -> tier (backend / frontend / infra), and role = service
  rule { source_labels = ["service"] target_label = "tier" regex = "(api|celery|worker).*" replacement = "backend" }
  rule { source_labels = ["service"] target_label = "tier" regex = "(frontend|web|next).*" replacement = "frontend" }
  rule { source_labels = ["service"] target_label = "role" replacement = "$1" }
  rule { source_labels = ["service"] action = "drop"
         regex = "(loki|grafana|alloy|prometheus|node-exporter|cadvisor|glitchtip).*" }
}

loki.source.docker "logs" {
  host          = "unix:///var/run/docker.sock"
  targets       = discovery.relabel.containers.output
  forward_to    = [loki.write.default.receiver]
  labels        = { env = "local", project = "myapp" }
}

// browser RUM receiver: the same-origin proxy posts here, written into Loki as logs
faro.receiver "frontend" {
  extra_log_labels = { env = "production", project = "myapp", kind = "faro" }
  server {
    listen_address       = "0.0.0.0"
    listen_port          = 12347
    cors_allowed_origins = ["*"]      // safe: only the server-side proxy posts here
    include_metadata     = true
  }
  output { logs = [loki.write.default.receiver] }
}

loki.write "default" {
  endpoint { url = "http://loki:3100/loki/api/v1/push" }
  external_labels = { agent = "alloy" }
}
```

Alloy here ships logs and RUM only. It does not scrape metrics; Prometheus does
that. On ECS the log collection is Fluent Bit via FireLens instead of the Alloy
docker tail, but the Faro receiver and the label taxonomy are the same. See
`instrumentation.md`.
